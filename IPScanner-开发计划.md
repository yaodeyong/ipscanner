# 内网 IP 管理系统开发计划（对标 Advanced IP Scanner）

## 1. 文档目的

- 以 `advanced_ip_scanner.exe` 的核心体验为参考，明确本项目的功能边界与实现路径。
- 将需求拆解为可执行任务，避免开发过程中目标漂移。
- 作为前后端、测试、部署共同遵循的统一基线文档。

## 2. 对标原则（参考 Advanced IP Scanner）

本项目不做逆向，不复制其私有实现，仅参考其成熟产品思路：

- **快速发现设备**：支持手动扫描与定时扫描，优先保证扫描结果可用性。
- **设备识别友好**：IP、MAC、厂商、主机名、在线状态集中展示。
- **冲突可追踪**：发现 IP-MAC 不一致时自动标记并记录历史。
- **管理闭环**：从“发现设备”到“分配/释放 IP”到“日志审计”全链路可追溯。
- **操作高效**：列表筛选、分页、批量导入导出，降低管理员日常操作成本。

## 3. 项目范围

### 3.1 V1 必做（MVP）

- React 单页应用（总览、冲突、设备详情、日志）。
- Node.js + Express REST API。
- MySQL 存储 IP 分配、设备信息、冲突记录、审计日志。
- 手动扫描 + 定时扫描（`node-cron`）。
- 实时 `check-ip` 冲突预检能力。
- JWT 鉴权（管理员角色）。

### 3.2 V1 暂不做

- 多租户与复杂权限体系。
- 高级报表与可视化大盘。
- 与 DHCP/交换机 API 深度联动。
- 分布式扫描节点与任务编排平台。

## 4. 技术架构

## 4.1 前端

- React + React Router + Axios + Ant Design。
- 状态管理：Context（V1）；如后续复杂度提高可切换 Redux Toolkit。
- 页面结构：顶部工具栏 + 左侧导航 + 主内容区。

## 4.2 后端

- Node.js + Express。
- ORM：Sequelize（推荐）或 Knex/原生 SQL（二选一，项目内保持一致）。
- 分层：`routes -> controllers -> services -> models/utils`。

## 4.3 数据库

- MySQL 8.x。
- 核心表：`ip_assignments`、`devices`、`conflicts`、`audit_log`、`oui`（可选）。

## 4.4 扫描模块

- Linux 优先：`arp-scan --localnet`（需权限）。
- Windows 兼容：`arp -a` + ping 补充探测。
- 统一结果格式后入库，避免平台差异污染业务层。

## 5. 功能模块与验收标准

## 5.1 IP 总览页

- 支持按子网、状态、MAC 模糊搜索。
- 列展示：IP、状态、MAC、厂商、备注、最后在线、操作。
- 分页稳定，接口支持 `page/pageSize`。

验收标准：

- 10k 以内 IP 记录可分页流畅查询（合理索引前提下）。
- 状态灯与后端 `status` 一致，不出现错色/错态。

## 5.2 新增/编辑 IP

- 表单校验 IP/MAC 格式。
- 输入 IP 触发 `GET /api/check-ip` 实时预检。
- 有冲突时禁止提交并显示明确原因。

验收标准：

- 非法 IP/MAC 不可提交。
- 冲突判定可复现且提示信息包含关键字段（在线状态、已分配 MAC、冲突类型）。

## 5.3 冲突列表

- 展示冲突 IP、冲突 MAC 集合、首次/最后发现时间、处理状态。
- 支持“标记解决”和“强制分配（需二次确认）”。

验收标准：

- 每次冲突变化有日志可查。
- 冲突解决后状态可追踪，不丢历史。

## 5.4 设备详情

- 按 MAC 聚合历史记录。
- 展示历史 IP 使用区间、厂商、最近在线状态。

验收标准：

- 设备历史链路完整，无断裂或重复覆盖异常。

## 5.5 操作日志

- 记录新增、编辑、释放、扫描触发等关键操作。
- 包含操作人、时间、对象、变更摘要。

验收标准：

- 任一写操作可在日志中追溯到对应记录。

## 6. API 规范（V1）

- `GET /api/ips`：分页筛选列表。
- `GET /api/ips/:ip`：IP 详情。
- `POST /api/ips`：新增分配。
- `PUT /api/ips/:ip`：更新信息。
- `DELETE /api/ips/:ip`：释放 IP。
- `GET /api/check-ip?ip=...`：预检在线与占用冲突。
- `POST /api/scan`：手动触发扫描。
- `GET /api/conflicts`：冲突列表。
- `PUT /api/conflicts/:id/resolve`：标记已解决。
- `GET /api/devices/:mac`：设备历史。
- `GET /api/logs`：审计日志查询。

统一响应建议：

- 成功：`{ code: 0, message: "ok", data }`
- 失败：`{ code: <业务码>, message: <错误信息>, details? }`

## 7. 数据模型约束（关键）

- `ip_assignments.ip_address`：唯一索引。
- `ip_assignments.status`：普通索引。
- `devices.mac`：主键。
- `conflicts.ip_address`：索引。
- `audit_log.created_at`：索引。

数据规范：

- MAC 入库前统一为大写冒号格式：`AA:BB:CC:DD:EE:FF`。
- IP 入库前做合法性校验与标准化。
- `details` 字段使用 JSON 字符串记录变更摘要。

## 8. 核心业务规则（不可偏离）

## 8.1 分配 IP 规则

1. 请求入参校验（IP、MAC、备注长度）。
2. 查询当前 IP 在线状态（短超时 ping + ARP）。
3. 查询数据库中该 IP 的已分配记录。
4. 若在线且在线 MAC 与请求 MAC 不一致，判定冲突，拒绝写入。
5. 若数据库已分配且 MAC 不一致，判定冲突，拒绝写入。
6. 否则写入或更新为 `assigned`。

## 8.2 扫描更新规则

1. 扫描得到在线 IP-MAC 列表。
2. 若数据库同 IP 绑定不同 MAC，写入 `conflicts` 并标记 IP `conflict`。
3. 若 IP 存在且 MAC 一致，更新 `last_online`。
4. 若 IP 不存在但在线，标记 `unknown`（待管理员认领）。

## 8.3 冲突解决规则

- 仅管理员可执行解决操作。
- 解决时必须记录 `resolved_at` 与 `resolution_note`。
- 强制分配需二次确认，且必须写入审计日志。

## 9. 安全与稳定性要求

- JWT 鉴权，所有写接口必须鉴权。
- 输入校验与输出转义，防注入/XSS。
- 扫描命令不接收未经校验的动态拼接参数。
- 扫描任务加“运行中锁”，避免并发重复扫描。
- `POST /api/scan` 增加限流（例如 30 秒内仅允许一次）。

## 10. 开发里程碑（建议）

## M1：基础骨架（1-2 天）

- 完成前后端脚手架。
- 数据库建表脚本与基础模型。
- JWT 登录鉴权与统一错误处理中间件。

交付物：

- 可运行的前后端工程 + SQL 初始化脚本。

## M2：IP 管理闭环（2-3 天）

- IP 总览页、增改删接口。
- `check-ip` 接口与前端实时预检联动。

交付物：

- 可完成 IP 分配、编辑、释放的闭环。

## M3：扫描与冲突（2-3 天）

- 手动扫描与定时扫描。
- 冲突识别、冲突列表、解决流程。

交付物：

- 可稳定发现并处理冲突。

## M4：设备与日志（1-2 天）

- 设备详情页（按 MAC 聚合历史）。
- 审计日志查询与筛选。
- 基础部署文档（Nginx + PM2 + MySQL）。

交付物：

- 功能完整的 V1 版本 + 部署说明。

## 11. 测试与验收清单

- 功能测试：增改删查、扫描、冲突解决、日志追溯。
- 异常测试：非法 IP/MAC、重复提交、扫描失败重试。
- 安全测试：鉴权绕过、SQL 注入、XSS 基础检查。
- 性能测试：分页查询、扫描结果入库耗时、并发接口响应。

上线前必须满足：

- 所有 P1 缺陷关闭。
- 扫描任务连续运行 24 小时无崩溃。
- 关键链路（分配、冲突、日志）可追溯。

## 12. 防偏机制（执行约束）

- 每个里程碑结束后进行一次“对标检查”：
  - 是否提升了设备发现与识别效率？
  - 是否强化了冲突发现与处理闭环？
  - 是否保证了操作可追溯？
- 未通过检查，不进入下一里程碑。
- 新增需求必须标注：是否属于 V1，若否进入 V1.1 待办池。
